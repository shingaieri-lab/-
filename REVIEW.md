# IS進捗管理ツール（ダンドリワーク）UXレビュー

レビュー日：2026-03-23

---

## このツールは何をするもの？

営業チームが「どの会社に連絡したか」「次はいつ連絡するか」を管理するツール。
AIが通話メモを読んで次の行動を提案したり、Googleカレンダーの空き時間を自動で探したりできる。

---

## 1. 使いやすさ（クリック数）

### ✅ うまくいっているところ

- リード新規登録：**3クリック**で完了
- 設定画面を開く：**3クリック**

### ❌ 改善が必要なところ

**候補日を探してメールを送るまでの流れが長すぎる（8〜10クリック）**

```
①候補日ページを開く
②検索条件を入力する
③「検索」ボタンを押す
④空き時間を3つ選ぶ
⑤「メールに使う」ボタンを押す　← ページが切り替わる
⑥メールページでもう一度リードを選ぶ　← ❌ さっき選んでいるのに！
⑦テンプレートを選ぶ
⑧「Gmail下書き保存」を押す
```

**⑥がムダ。** 候補日ページで選んだリード情報をメールページに引き継ぐべき。

## 2. 見た目の一貫性

### ❌ AIページだけ見た目が全然違う

| ページ | 背景・テキスト |
|--------|--------------|
| ダッシュボード・リード管理など | 白背景＋緑文字 |
| **AIページだけ** | 真っ暗な紺色背景＋白文字 |

「AIアシスタント」タブを押した瞬間に別のアプリのような見た目になる。
初めて使うユーザーが「バグ？」と感じる可能性がある。
他のページと同じ白系背景に統一、またはわずかな差にとどめるのが望ましい。

### ❌ ナビのラベルが小さすぎる

| 場所 | フォントサイズ |
|------|-------------|
| デスクトップ サイドバー | **9.5px** |
| スマホ ボトムナビ | **8px** |

一般的なWebの最小推奨は12px。8pxはほぼ読めない大きさ。
アイコンで判断させる意図であれば、ラベルを非表示にするか、最低11〜12pxに拡大すべき。

## 3. コードの問題（今後の開発に影響）

### ❌ Gmail下書き保存のコードが2か所に重複（約80行）

`EmailPage` の `saveGmailDraft` と `AIPage` の `saveAiGmailDraft` がほぼ同じ内容。

- **問題：** バグ修正・仕様変更のとき2か所を同時に直さないと片方だけ古くなる
- **対策：** 共通関数（例：`saveGmailDraftCommon`）を1つ作り、両ページから呼ぶ

### ❌ 祝日リストが2か所に重複定義

```javascript
// 行178（グローバル）
const JP_HOLIDAYS = new Set(["2025-01-01", ...])

// 行695（AIPage内）
const JP_HOL = new Set(["2025-01-01", ...])  ← 全く同じ内容
```

- **問題：** 2027年以降の祝日追加時、片方を更新し忘れると誤動作する
- **対策：** グローバルの `JP_HOLIDAYS` を AIPage でも参照するだけでよい

### ❌ メールテンプレートの初期値が2か所で微妙にズレている

| 場所 | 変数名 |
|------|--------|
| 行211（グローバル） | `{{担当者名}}` |
| 行297（EmailPage内） | `{{担当者苗字}}` （内容も少し違う） |

どちらが正しいか判断しづらく、将来の混乱のもとになる。

### ❌ 単一HTMLファイル 4550行

全ロジック・スタイル・テンプレートが1ファイルに集中しており、機能追加・バグ修正の難易度が高い。
長期的にはコンポーネントごとにファイル分割を検討する。

## 4. セキュリティの問題

### 🔴 パスワードが暗号化なしで保存されている

`server.js` 41行目：

```javascript
// ❌ 現状：パスワードをそのまま比較している
const account = accounts.find(
  a => a.id === id && a.password === password
);
```

パスワードが Vercel KV に**平文で保存**されている。
データベースの内容が漏れた場合、全ユーザーのパスワードがそのまま読まれる。

- **対策：** `bcryptjs` などでハッシュ化して保存・比較する（Webサービスの標準対策）

---

## 5. AIページ：解析後の導線が不明

解析→保存した後、次に何をすればいいかの案内がない。

**現状：**
```
解析する → 保存完了！ ← ここで終わり
```

**改善案：**
```
解析する → 保存完了！
           → 「📧 フォローメールを送る」ボタン
           → 「📅 商談候補日を探す」ボタン
```

---

## 6. 設定の初回セットアップが難しすぎる

Google Calendar 連携を開始するまでの手順：

1. Google Cloud Console でプロジェクト作成
2. Calendar API を有効化
3. APIキーを取得
4. OAuth クライアントID を作成
5. 承認済みオリジンの設定
6. 各担当者のカレンダーID を収集
7. アプリの設定画面に全部入力

現状は文字の説明だけで案内しており、IT知識のないユーザーには高いハードル。

- **改善案：** ステップ1〜7を順に案内する「ウィザード形式」の画面を作る

---

## 7. 確認ダイアログがブラウザのポップアップ

削除確認に `window.confirm`（ブラウザ標準のグレーのポップアップ）を使用。
アプリのデザインと統一感がなく、ユーザーが戸惑う。

- **改善案：** アプリ内モーダル（ダイアログ）で確認を取る

---

## 8. 良い点

- KPIカードの視認性が高く、数値が一目でわかる
- LeadCombobox のインクリメンタルサーチ（入力しながら絞り込む）が使いやすい
- モバイル対応がされていてボトムナビも直感的
- エラーバウンダリが実装されており、予期しないエラーに強い
- AI解析結果（興味度・トークポイント・ネクストアクション日）が実用的
- 候補日検索の曜日・バッファ・対象外時間帯の設定が柔軟
