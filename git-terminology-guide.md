# Gitの用語をやさしく解説

> **Git（ギット）とは？**
> ファイルの変更履歴を管理してくれるツールです。
> 「いつ・誰が・何を変えたか」を記録し、過去の状態に戻したり、複数人で同時に作業したりできます。

---

## リポジトリ（Repository）

**一言で言うと：プロジェクトの「保管箱」**

コード・画像・設定ファイルなど、プロジェクトに関するすべてのファイルをまとめて管理する場所です。
よく「リポ」と略して呼ばれます。

| 種類 | 場所 | イメージ |
|------|------|---------|
| ローカルリポジトリ | 自分のパソコンの中 | 手元の下書きノート |
| リモートリポジトリ | GitHubなどのサーバー上 | クラウドに保存された清書ノート |

---

## コミット（Commit）

**一言で言うと：変更内容の「セーブ」**

ゲームのセーブポイントと同じイメージです。
「この状態を記録しておく」という操作で、後からその時点に戻ることができます。

```bash
# 例：「ログイン機能を追加した」という内容でセーブする
git commit -m "ログイン機能を追加"
```

> **ポイント：** コミットするときは、何を変えたかわかるメッセージを書きましょう。
> 後で見返したとき、自分も他の人も内容が把握できます。

---

## ステージ（Stage）

**一言で言うと：コミットに含めるファイルを選ぶ「準備エリア」**

複数のファイルを変更したとき、「これとこれだけをまとめてセーブしたい」と選ぶ場所です。

```
【変更したファイル】         【ステージ（準備エリア）】       【コミット（セーブ済み）】

  ファイルA ─── add ───→    ファイルA  ─── commit ───→   ファイルAの変更を記録
  ファイルB ─── add ───→    ファイルB
  ファイルC（まだ準備しない）
```

```bash
# 特定のファイルだけステージに追加する
git add ファイルA

# 変更したすべてのファイルをステージに追加する
git add .
```

---

## プッシュ（Push）

**一言で言うと：自分のセーブをサーバーに「アップロード」する**

コミット（セーブ）は最初、自分のパソコンの中にしかありません。
プッシュすることで、GitHub などのサーバーにも送られ、他の人と共有できます。

```
自分のPC（ローカル）  ──── push ────→  GitHub（リモート）
```

```bash
git push origin main
#         ↑      ↑
#      送り先  ブランチ名（後述）
```

---

## プル（Pull）

**一言で言うと：サーバーの最新情報を自分のPCに「ダウンロード」する**

チームで作業していると、他のメンバーが変更をプッシュしていることがあります。
プルすることで、サーバーの最新状態を自分のPCに取り込めます。

```
GitHub（リモート）  ──── pull ────→  自分のPC（ローカル）
```

```bash
git pull origin main
```

---

## フェッチ（Fetch）

**一言で言うと：サーバーの変更を「確認だけ」する**

プルと似ていますが、自分のファイルには反映させずに、サーバー側に何か変更があるか確認するだけの操作です。
「ダウンロードする前に中身を確認する」イメージです。

```bash
git fetch origin main
```

> **プルとの違い：**
> - `fetch`：情報の取得のみ。手元のファイルは変わらない
> - `pull`：情報の取得＋手元のファイルに反映まで行う

---

## ブランチ（Branch）

**一言で言うと：本番に影響しない「作業用の分岐」**

木の幹（本番のコード）から枝（作業用のコピー）を分けて、そこで自由に作業できます。
本番を壊す心配なく、新機能の追加やバグ修正を試せます。

```
main（本番）  ───●───●───────────●───  ← 本番はそのまま動いている
                  |               ↑
                  └───●───●───●──┘  ← 枝で作業して、完成したら本番に合流
                  （新機能ブランチ）
```

```bash
# 新しいブランチを作って移動する
git switch -c 新機能の名前

# 既存のブランチに移動する
git switch ブランチ名
```

---

## マージ（Merge）

**一言で言うと：枝の作業を本番に「合流」させる**

ブランチで完成した変更を、本番（mainブランチなど）に取り込む操作です。

```bash
# mainブランチに移動してから
git switch main

# 作業ブランチの内容を取り込む
git merge 作業ブランチ名
```

---

## コンフリクト（Conflict）

**一言で言うと：変更の「ぶつかり合い」**

同じファイルの同じ行を、2人が別々に変更してマージしようとするときに起きます。
どちらの変更を採用するか、手動で選んで解決する必要があります。

```
<<<<<<< HEAD
Aさんが書いた内容
=======
Bさんが書いた内容
>>>>>>> feature-branch
```

上の `<<<<<<<` と `>>>>>>>` の間が「ぶつかっている部分」です。
どちらかを残す（または両方を合わせる）よう編集して解決します。

---

## プルリクエスト（Pull Request / PR）

**一言で言うと：「この変更をマージしてください」という申請**

GitHubなど上で行う操作で、チームのメンバーに変更内容を確認してもらい、問題なければ本番に取り込んでもらえます。
「レビュー → 承認 → マージ」という流れで安全に作業を進められます。

---

## スタッシュ（Stash）

**一言で言うと：作業を一時的に「棚上げ」する**

作業の途中でブランチを切り替えたいけど、まだコミットしたくないというときに使います。
一時退避させて後で取り出せます。

```bash
# 作業を棚上げする
git stash

# 棚上げした作業を元に戻す
git stash pop
```

---

## チェックアウト（Checkout）

**一言で言うと：ブランチの切り替えや、過去のファイルを取り出す操作**

以前は `git checkout` がブランチ切り替えにも使われていましたが、
現在は `git switch`（ブランチ切り替え）と `git restore`（ファイルの復元）に役割が分かれています。

```bash
git switch ブランチ名    # ブランチを切り替える（新しい書き方）
git checkout ブランチ名  # 同じ意味（昔からある書き方）
```

---

## 基本的な作業の流れ（まとめ）

```
1. ブランチを作る
   └─ git switch -c 作業ブランチ名

2. コードを変更する
   └─ ファイルを編集する

3. ステージに追加する
   └─ git add .

4. コミットする（セーブ）
   └─ git commit -m "何を変えたか"

5. プッシュする（サーバーにアップ）
   └─ git push origin 作業ブランチ名

6. プルリクエストを作る
   └─ GitHubの画面から操作

7. レビューを受けてマージする
   └─ 承認されたら本番に反映完了！
```

---

> **困ったときは：**
> - `git status` → 今の状態を確認できる
> - `git log` → コミットの履歴を見られる
> - `git diff` → 変更した内容の差分を確認できる
